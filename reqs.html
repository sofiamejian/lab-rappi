Lab 4 — Rappi Ecosystem: Real-time Delivery Tracking

1. Contexto

En este laboratorio, continuaremos con el ecosistema inspirado en Rappi desarrollado en el Lab 3. El objetivo es extender el sistema agregando seguimiento en tiempo real de entregas utilizando mapas interactivos, geolocalización y comunicación en tiempo real mediante Supabase Realtime (Broadcast).

El sistema ya cuenta con un backend (Express + TypeScript), tres clientes web independientes (Consumer App, Store App, Delivery App) construidos con React + Vite, una base de datos relacional en Supabase y deploy en Vercel.

Nuevas tecnologías a integrar:
Leaflet / React-Leaflet PostGIS (GEOGRAPHY) Supabase Realtime (Broadcast) KeyboardEvent API

2. Requerimientos Funcionales

2.1 Delivery App (Rappi)

  • Cuando un repartidor acepte un pedido, el sistema debe mostrar un mapa interactivo con la posición actual del repartidor y el punto de recogida del pedido (tienda).
  • El repartidor debe poder moverse en el mapa utilizando las teclas del teclado:
    • ArrowUp — mover hacia arriba (incrementar latitud)
    • ArrowDown — mover hacia abajo (decrementar latitud)
    • ArrowLeft — mover hacia la izquierda (decrementar longitud)
    • ArrowRight — mover hacia la derecha (incrementar longitud)
  • Cada movimiento debe actualizar la posición del repartidor en la base de datos y emitir un evento en tiempo real vía Supabase Broadcast para que el consumidor vea el movimiento.
  • Cuando el repartidor se encuentre a menos de 5 metros del lugar de recogida, el sistema debe cambiar el estado del pedido a "Entregado" y notificar al consumidor.

2.2 Consumer App

  • Al crear un pedido, el consumidor debe poder seleccionar una ubicación de entrega en el mapa (click en el mapa para definir el punto).
  • Cuando un repartidor acepte su pedido, el consumidor debe ver un mapa en tiempo real con la posición del repartidor actualizándose en vivo.
  • Cuando el repartidor llegue (distancia < 5 metros), el consumidor debe recibir una notificación visual indicando que el repartidor ha llegado.

2.3 Store App

  • La tienda debe poder ver el estado actualizado de sus pedidos: Creado, En entrega, Entregado.

3. Requerimientos No Funcionales

  • Se debe mantener la arquitectura de 3 clientes web independientes + 1 backend.
  • El backend debe exponer una API REST.
  • La comunicación en tiempo real debe usar Supabase Realtime (Broadcast).
  • Los datos geoespaciales deben almacenarse usando el tipo GEOGRAPHY(POINT, 4326) de PostGIS.
  • Los estados del pedido deben definirse como un enum en TypeScript.
  • El cálculo de distancia debe realizarse con ST_DWithin de PostGIS.

4. Modelo de Datos

Se deben agregar las siguientes columnas a la tabla orders:

Columna Tipo Restricción Descripción

CREATE EXTENSION IF NOT EXISTS postgis;

4.3 Entendiendo GEOGRAPHY(POINT, 4326)

GEOGRAPHY(POINT, 4326) es un tipo de dato de PostGIS para almacenar coordenadas geográficas:

  • POINT = un punto con coordenadas (longitud, latitud)
  • 4326 = sistema de coordenadas estándar GPS (WGS84), el mismo que usa Google Maps, Leaflet, etc.
Orden importante: PostGIS usa el orden (longitud, latitud), NO (latitud, longitud).
Por ejemplo, para Cali, Colombia: ST_MakePoint(-76.53, 3.45) donde -76.53 es la longitud y 3.45 es la latitud.

4.4 Insertar un punto geográfico

-- Insertar una ubicación como GEOGRAPHY
INSERT INTO orders (id, consumer_id, store_id, status, destination)
VALUES (
  gen_random_uuid(),
  'uuid-consumer',
  'uuid-store',
  'Creado',
  ST_SetSRID(ST_MakePoint(-76.53, 3.45), 4326)
);

4.5 Actualizar la posición del repartidor

-- Actualizar posición del repartidor en cada movimiento
UPDATE orders
SET delivery_position = ST_SetSRID(ST_MakePoint(-76.532, 3.451), 4326)
WHERE id = 'uuid-order';

4.6 Consultar por cercanía (distancia en metros)

-- Verificar si el repartidor está a menos de 5 metros del destino
SELECT
  id,
  ST_Distance(delivery_position, destination) AS distance_meters
FROM orders
WHERE id = 'uuid-order'
  AND ST_DWithin(
    delivery_position,
    destination,
    5  -- distancia en metros
  );

ST_DWithin retorna true si la distancia entre dos puntos geográficos es menor o igual al valor dado en metros. Esto es lo que usaremos para detectar la llegada del repartidor.

4.7 Ejemplo general de consulta por cercanía

-- Buscar registros dentro de un radio de 1000 metros
SELECT *
FROM lugares
WHERE ST_DWithin(
  ubicacion,
  ST_SetSRID(ST_MakePoint(-76.53, 3.45), 4326),
  1000  -- distancia en metros
);

5. Enum de Estados del Pedido

Los estados del pedido deben ser definidos como un enum en TypeScript y utilizarse tanto en el backend como en la validación: 

enum OrderStatus {
  CREATED    = "Creado",
  IN_DELIVERY = "En entrega",
  DELIVERED  = "Entregado",
}
Estado Cuándo se asigna Quién lo cambia
Creado Cuando el consumidor crea el pedido Sistema (automático)
En entrega Cuando un repartidor acepta el pedido Repartidor
Entregado Cuando el repartidor llega al destino (< 5 metros) Sistema (automático)

6.2 Flujo de tiempo real con Supabase Broadcast

// Delivery App: emitir posición al mover con teclas
const channel = supabase.channel(`order:${orderId}`);

channel.send({
  type: 'broadcast',
  event: 'position-update',
  payload: { lat: 3.451, lng: -76.532 }
});

// Consumer App: escuchar posición en tiempo real
const channel = supabase.channel(`order:${orderId}`);

channel.on('broadcast', { event: 'position-update' }, (payload) => {
  // Actualizar marcador del repartidor en el mapa
  setDeliveryPosition(payload.payload);
}).subscribe();

6.3 Manejo de teclas con throttle (Delivery App)

El movimiento visual en el mapa debe ser inmediato (actualiza el estado local al instante), pero el PATCH a la base de datos debe hacerse con un throttle de 1 segundo: sin importar cuántas teclas se presionen, la API se llama como máximo una vez por segundo con la posición más reciente. 

const STEP = 0.00005; // ~5 metros aprox.

// useRef para el throttle: guarda si ya hay una llamada programada
const throttleRef = useRef<ReturnType<typeof setTimeout> | null>(null);
// useRef para capturar siempre la posición más reciente dentro del timeout
const pendingPosition = useRef(position);

useEffect(() => {
  const handleKeyDown = (e: KeyboardEvent) => {
    let { lat, lng } = position;

    switch (e.key) {
      case 'ArrowUp':    lat += STEP; break;
      case 'ArrowDown':  lat -= STEP; break;
      case 'ArrowLeft':  lng -= STEP; break;
      case 'ArrowRight': lng += STEP; break;
      default: return;
    }

    // 1. Mover el marcador en el mapa de inmediato
    setPosition({ lat, lng });
    pendingPosition.current = { lat, lng };

    // 2. Si ya hay un ciclo de 1 seg activo, no hacer nada más
    if (throttleRef.current) return;

    // 3. Abrir un nuevo ciclo: esperar 1 seg y luego enviar la posición más reciente
    throttleRef.current = setTimeout(() => {
      updatePosition(pendingPosition.current); // API call + broadcast
      throttleRef.current = null; // liberar para el siguiente ciclo
    }, 1000);
  };

  window.addEventListener('keydown', handleKeyDown);
  return () => {
    window.removeEventListener('keydown', handleKeyDown);
    if (throttleRef.current) clearTimeout(throttleRef.current);
  };
}, [position]);
Debounce vs Throttle
  • Debounce: espera a que el usuario deje de presionar para disparar. Si sigue moviéndose indefinidamente, la llamada nunca se hace.
  • Throttle: dispara una vez por intervalo de tiempo fijo, sin importar cuántas teclas se presionen. Ideal para movimiento continuo.

pendingPosition es un useRef para que el setTimeout siempre lea la coordenada más reciente al dispararse, evitando el problema de stale closure.
